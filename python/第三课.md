# 第 3 课：L6 Memory 与 Checkpoint — 课程记录

## 本课目标

理解多轮对话不是「模型记得」，而是 LangGraph checkpoint 保存了状态并在下一轮恢复。

## 核心对比

```
无记忆：
  调用1 ──→ agent.stream(...) → 查到了 Frank Harris 发票
  调用2 ──→ agent.stream(...) → 不记得，重新查表结构，反问用户

有记忆（checkpointer + thread_id）：
  调用1 ──→ agent.stream(..., thread_id="1") → 结果存入 InMemorySaver
  调用2 ──→ agent.stream(..., thread_id="1") → 取出历史 messages 拼给模型
              → 模型直接关联上一轮的发票 ID，查曲目标题
```

## 关键修正记录

### 修正 1：记忆机制的本质

- ❌（有人会误以为）LLM 自己记住了历史
- ✅ 是 `checkpointer` 保存了历史 messages，下一轮调用时**重新塞给模型**
- 证据：L1-L5 用同样的 LLM，但没有记忆 → 说明记忆来自工程层，不是模型层

### 修正 2：`InMemorySaver` 的生产局限

- `InMemorySaver()` 保存在当前 Python 进程内存中
- 内核重启 → 记忆全部丢失
- 生产环境必须用持久化方案

## Checkpoint 的两个要素

```python
# 1. create_agent 时指定 checkpointer
agent = create_agent(
    ...,
    checkpointer=InMemorySaver(),
)

# 2. stream 时指定 thread_id（同一个 id = 同一条对话线）
for step in agent.stream(
    {"messages": [...]},
    {"configurable": {"thread_id": "1"}},   # ← 会话标识
    context=RuntimeContext(db=db),
    stream_mode="values",
):
```

**`thread_id` 的本质**：它是 checkpoint 存储的 key。同一个 key → 读取/写入同一段历史。不同的 key → 隔离的对话线。

## Checkpointer 方案对比

| 方案 | 存储位置 | 重启后恢复 | 适用场景 |
|---|---|---|---|
| `InMemorySaver` | 进程内存 | ❌ | 学习开发 |
| `SqliteSaver` | 本地 SQLite 文件 | ✅ | 单机生产 |
| `PostgresSaver` | PostgreSQL | ✅ | 分布式生产 |
| 自定义 `BaseCheckpointSaver` | Redis / MySQL / S3 | ✅ | 定制需求 |

## 多租户注意事项

- 不同用户的对话必须使用**不同的 `thread_id`**
- 简单方案：`f"user_{user_id}_session_{session_id}"`
- 如果两个用户共用 `thread_id="1"` → 上下文污染，A 看到 B 的对话

## 与 L9 HITL 的关联

L6 的 checkpoint 和 L9 的中断恢复使用**同一套底层机制**：
- L6：保存 messages → 下一轮恢复上下文
- L9：保存 messages + 中断点 → 人工审批后从断点恢复执行
- 二者的共同基础：`checkpointer` + `thread_id`

## 术语速查

| 术语 | 含义 |
|---|---|
| `InMemorySaver` | LangGraph 内置的进程内存 checkpointer |
| `checkpointer` | create_agent 参数，指定状态保存机制 |
| `thread_id` | 会话线标识 key，同一 id 共享历史 |
| Checkpoint | 每次 agent 步骤后的状态快照（messages + 执行位置） |
| 上下文污染 | 不同用户的对话用同一 thread_id 导致历史错乱 |
| `SqliteSaver` / `PostgresSaver` | 生产级持久化 checkpointer |
