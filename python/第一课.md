# 第 1 课：L1 快速搭建 SQL Agent — 课程记录

## 本课目标

理解一个最小 agent 是怎么装起来并跑起来的，建立 `create_agent` 装配入口的完整认知。

## 核心链路

```
用户问题
  ↓
{"messages": question}  ← 输入格式
  ↓
create_agent( model, tools, system_prompt, context_schema )
  ↓
agent.stream( ..., context=RuntimeContext(db=db), stream_mode="values" )
  ↓
模型决定是否调用 execute_sql
  ↓
  ├─ 不调用 → 直接回答
  └─ 调用 → get_runtime(RuntimeContext).context.db → db.run(query)
              ↓
            工具结果返回给模型（ToolMessage）
              ↓
            模型给出最终回答（AiMessage）
```

## 关键修正记录

### 修正 1：`@dataclass` 的作用

- ✅ `@dataclass` 是 Python 内置装饰器，自动生成 `__init__`、`__repr__`、`__eq__` 等方法
- ❌ 和 `@tool`、agent 没有关系，只是一个普通数据容器的语法糖
- 等价于手动写 `class RuntimeContext: def __init__(self, db): self.db = db`

### 修正 2：`get_runtime()` 的返回值结构 ⭐ 核心概念

```
get_runtime(RuntimeContext)  → 返回 LangGraph 包装对象
                                   ↓
                            runtime.context   → 你定义的 RuntimeContext 实例
                                   ↓
                            runtime.context.db → 你注入的 SQLDatabase 对象
```

- ❌ **误以为** `runtime.db` 直接可取
- ✅ **实际是** `runtime.context.db`，因为 `runtime` 是包装对象，`runtime.context` 才是你的数据

### 修正 3：`context_schema` 不是可选的

- ❌ 「缺失 schema 影响不大」
- ✅ `context_schema=RuntimeContext` 在 create 阶段声明上下文结构，缺失则 `get_runtime()` 会报错
- `context=RuntimeContext(db=db)` 在 invoke/stream 阶段注入实际数据
- **两者都必不可少**

### 修正 4：`for step in agent.stream()` 不是异步流

- ❌ 以为 stream 是「断断续续发送数据」
- ✅ `stream_mode="values"` 是**每完成一步 agent 执行就产出一个状态快照**，for 循环同步阻塞等待每一步完成
- `stream_mode="messages"` 才是真正的 token 级流式输出

### 三种调用模式对比

| 方式 | 返回粒度 | 适合场景 |
|---|---|---|
| `agent.invoke(...)` | 一次性返回最终结果 | 只要最终答案 |
| `agent.stream(stream_mode="values")` | 每完成一步返回一次 | 调试、日志、观察推理过程 |
| `agent.stream(stream_mode="messages")` | 逐 token / 逐消息块 | 聊天 UI、实时展示 |

## 四种 Message 类型

来自 Cell 8 的观察：

```
Human Message  → 用户问题
Ai Message     → 模型思考 + 提出 Tool Calls
Tool Message   → 工具执行结果
Ai Message     → 模型基于工具结果给出最终回答
```

## 工程要点

### 工具层硬约束 vs Prompt 软约束

| | Prompt 软约束 | 工具层硬约束 |
|---|---|---|
| 方式 | 在 system prompt 里写规则 | 在工具函数内部做校验 |
| 可靠性 | 模型可以忽略 | 强制执行 |
| L1 示例 | SYSTEM_PROMPT 写了只读规则 | 无（纯透传） |
| 我的项目 | - | 关键词拦截 DROP/INSERT 等 |
| 结论 | 不够安全 | 更可靠，但仍然有局限（如 SQL 关键字绕过） |

### L1 的限制

1. **无记忆**：每次调用之间不保留状态，同一问题问两次会重新查询
2. **无参数校验**：`execute_sql(query: str)` 接收裸字符串，无格式/类型/范围约束
3. **纯 Prompt 安全**：只读规则写在 system prompt 里，模型可以忽略
4. **错误处理粗糙**：`return f"Error: {e}"` 未区分错误类型

## 课后练习要点

### 练习 2：`context={"db": db}` vs `context=RuntimeContext(db=db)`

- 两种写法都可运行
- dict 写法是 LangGraph 的兼容语法，直接按字段名匹配
- dataclass 写法更类型安全，有 IDE 补全和类型检查

### 关于模型「主动性」的观察

当用户问题模糊（如 "TRY YOUR OWN QUERY HERE"），模型仍会主动调用工具探索数据库。这说明：**模型有自主决策能力，但并不安全**——它可能执行你没预料到的查询。

## 术语速查

| 术语 | 含义 |
|---|---|
| `create_agent` | LangGraph 的 agent 装配 API，不是普通函数而是「声明式配置入口」 |
| `RuntimeContext` | 运行时上下文的数据结构声明，用于向工具注入资源（如 db） |
| `context_schema` | create 阶段声明上下文结构的参数 |
| `get_runtime(T)` | 工具内部获取当前运行时上下文的函数 |
| `stream_mode` | agent 输出模式控制参数 |
| `tool` | 将普通函数包装成模型可调用的工具 |
| `ReAct` | 模型思考→调用工具→接收结果的循环模式 |
